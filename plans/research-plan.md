# 任务质量仪表盘研究任务规划

## 1. 规划边界

本计划用于调度“spec-executor 任务质量仪表盘”项目的**研究任务**。规划输入为 `inputs/project-brief.md` 与 `docs/PRD.md`：项目需要先研究用户需要看哪些质量信号，再设计数据模型与交互，最后实现 CLI 导出与 Markdown 报告；研究阶段只回答“用户需要看哪些质量信号、如何度量、从哪里取证据”，为后续 design-planning 提供确认过的输入。

本任务只产出 `plans/research-plan.md`，**不执行研究、不给出研究结论、不修改 task package、不修改源码 / schema / 迁移 / 运行时脚本 / 文档正文**。本计划描述后续研究任务应当如何被拆分、排序、并行、验收与停止，供 spec-executor 调度；具体研究结论由后续研究任务在执行阶段产出，本任务不预填任何结论。

来源需求映射：
- `docs/PRD.md` FR-001：研究规划任务输出研究计划。
- `docs/PRD.md` FR-004：所有计划必须包含任务、依赖、验收和并行策略。
- `inputs/project-brief.md`：产品链路为“研究质量信号 → 设计数据模型与交互 → 实现 CLI 导出与 Markdown 报告”，研究任务只覆盖第一段链路。

## 2. 研究任务拆分

以下为可独立调度、可独立验收的研究任务。每个任务只产出研究记录（问题、方法、证据、开放问题），不产出结论性判断以外的实现内容，且不越出 brief/PRD 已确认范围臆造产品事实。

| 任务 ID | 研究任务名称 | 研究问题（待回答，不在本计划回答） | 建议产出区域 | 研究职责 |
| --- | --- | --- | --- | --- |
| RS-01 | 用户与使用场景研究 | 谁在看仪表盘（角色）？他们在什么场景、为什么决策而看？ | 后续研究记录（如 `docs/research/users-and-scenarios.md`） | 明确目标用户角色、使用场景、决策诉求，界定“需要看哪些质量信号”的问题空间。 |
| RS-02 | 质量信号清单研究 | 用户需要看哪些任务质量信号？哪些是核心、哪些是次要？ | 后续研究记录（如 `docs/research/quality-signals.md`） | 依据 RS-01 的角色与场景，收集并归类候选质量信号，标注优先级与开放问题。 |
| RS-03 | 指标口径与度量方法研究 | 每个质量信号如何度量、如何计算、边界与缺失数据如何界定？ | 后续研究记录（如 `docs/research/metrics-definitions.md`） | 为 RS-02 的信号研究度量口径、计算方式、阈值/分层思路与缺失数据处理方向。 |
| RS-04 | 证据来源与数据可得性研究 | 每个信号/指标的证据从哪里取？spec-executor 现有产物是否可得？ | 后续研究记录（如 `docs/research/evidence-sources.md`） | 调研数据来源、可得性、采集成本与可信度，识别不可得或需替代的信号。 |
| RS-05 | 呈现形态与交互期望研究 | 用户期望以何种视图/报告形态查看质量信号（含空状态、钻取）？ | 后续研究记录（如 `docs/research/presentation-expectations.md`） | 调研展示形态、CLI 导出与 Markdown 报告的用户期望，为 design-planning 交互设计提供输入。 |
| RS-06 | 研究综合与移交验收 | 上述研究是否完整、一致、可作为 design-planning 的确认输入？ | 后续研究综合记录（如 `docs/research/research-summary.md`） | 汇总 RS-01..RS-05，做一致性与完整性验收，列出确认项、开放问题与阻塞项，形成 design-planning 移交包。 |

> 说明：上表“研究问题”列为**待后续研究任务回答的问题**，本计划不回答、不预设答案；“建议产出区域”为调度建议路径，最终路径由执行阶段确定。

## 3. 依赖 DAG（依赖关系）

| 依赖 | 原因（消费的上游输出） |
| --- | --- |
| RS-01 -> RS-02 | 信号清单必须基于已界定的用户角色与使用场景，避免脱离用户凭空罗列信号。 |
| RS-02 -> RS-03 | 指标口径研究必须针对已确认的候选质量信号展开。 |
| RS-02 -> RS-04 | 证据来源研究必须针对已确认的候选信号定位数据出处。 |
| RS-03 -> RS-06 | 综合验收需要指标口径与缺失数据方向的研究结果。 |
| RS-04 -> RS-06 | 综合验收需要证据可得性研究结果以判定信号是否落地。 |
| RS-01 -> RS-05 | 呈现形态研究需要用户角色与场景作为期望来源。 |
| RS-05 -> RS-06 | 综合验收需要呈现与交互期望的研究结果作为 design 移交输入。 |
| RS-02 -> RS-05 | 呈现形态需要知道要展示的核心信号，避免视图脱离信号清单。 |

DAG 节点：RS-01、RS-02、RS-03、RS-04、RS-05、RS-06。
环路检查：无环；RS-06 为唯一最终综合与验收节点。

## 4. 并行策略（可并行批次）

| 批次 | 可并行任务 | 调度条件 | 并行注意事项 |
| --- | --- | --- | --- |
| R1 | RS-01 | 读取 `inputs/project-brief.md` 与 `docs/PRD.md` 后启动。 | 用户与场景先行，避免下游信号研究脱离用户诉求。 |
| R2 | RS-02、RS-05 | RS-01 通过单任务验收后启动。 | RS-02 研究信号清单、RS-05 研究呈现期望；两者可并行，但须共用 RS-01 的角色/场景口径，命名与角色划分保持一致。 |
| R3 | RS-03、RS-04 | RS-02 通过单任务验收后启动。 | RS-03 研究度量口径、RS-04 研究证据来源；两者针对同一信号清单并行，须对信号命名与编号保持一致，避免口径与来源对不上。 |
| R4 | RS-06 | RS-03、RS-04、RS-05 全部通过单任务验收后启动。 | 只做综合、一致性检查与移交验收，不补做新的原始研究。 |

并行安全约束：同一批次内任务不得共享未决问题定义、信号命名、文件与状态，且不得存在直接/间接依赖路径；并行产出在进入下一批次前须做信号命名与编号一致性对齐。

## 5. 单任务验收标准（验收）

每个研究任务的验收必须满足：
- 只产出各自职责范围内的研究记录文件，不修改 task package、源码、schema、迁移、运行时脚本或其他任务的产出。
- 明确研究问题、研究方法/来源、已获得的证据，以及**未解决的开放问题**；无法确认的内容必须记录为开放问题或阻塞项，不得伪装成已确认事实。
- 与上游研究任务的角色划分、信号命名、编号口径保持一致；冲突时停止并要求修订上游研究。
- 每条被采纳的质量信号/指标可追溯到用户场景（RS-01）与证据来源（RS-04），不臆造 brief/PRD 之外的产品事实。
- 研究记录只描述“研究所得与开放问题”，不越界产出数据模型、交互设计或实现方案（这些属于 design-planning 与后续阶段）。

## 6. 全局验收（RS-06 综合与移交验收）

RS-06 依赖 RS-01..RS-05，须确认：
- “用户与场景 → 质量信号清单 → 指标口径 → 证据来源 → 呈现期望”研究链路完整，覆盖 brief 中“研究用户需要看哪些质量信号”的问题空间。
- 每个候选质量信号都有：所属用户场景、度量口径方向、证据来源判定（可得 / 不可得 / 需替代）、呈现期望，或被显式标注为开放/阻塞。
- 研究记录之间信号命名、编号、角色划分一致，无隐藏依赖、循环依赖或重复研究。
- 产出一份 design-planning 移交包：确认项清单、开放问题清单、阻塞项清单，且不包含任何设计或实现决策。
- 全程未执行超出研究范围的活动，未修改 task package 或非研究记录文件。

## 7. 停止条件（阻塞条件，停止）

任一研究任务遇到以下情况必须**停止**并记录阻塞原因，不得继续臆造内容：
- `inputs/project-brief.md` 或 `docs/PRD.md` 缺失，或其内容不足以界定该研究任务的问题空间。
- 需要引入 brief/PRD 未确认的用户角色、质量信号、指标口径、证据来源、报告/展示形态或运行环境事实，且无可信来源可依据。
- 上游研究任务未完成、验收失败，或上游产出与当前任务的信号命名/角色口径冲突。
- 证据来源不可得且无可接受替代来源，导致某信号无法研究其可得性（记录为阻塞并交由 RS-06 汇总）。
- 任务被要求执行研究以外的活动，或需要修改 task package、源码、schema、迁移、运行时脚本或其他非研究记录文件。
- 研究结论无法在不越界（不进入数据模型/交互/实现设计）的前提下表达。

出现停止条件时，任务应保留已完成的部分研究记录、明确标注阻塞点，并将阻塞项上报至 RS-06 综合验收，由调度方决定修订上游或降级信号范围。

## 8. 执行顺序（拓扑序）

1. RS-01（用户与场景）
2. 并行：RS-02（信号清单）、RS-05（呈现期望，依赖 RS-01/后续对齐 RS-02 命名）
3. 并行：RS-03(指标口径)、RS-04(证据来源)
4. RS-06（综合与移交验收）

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 用户角色/场景界定不足（RS-01 不充分） | 信号清单脱离真实诉求 | 停止 RS-02 并回写 RS-01 开放问题；不臆造用户。 |
| 信号命名在并行任务间漂移 | 口径与证据来源对不上 | R2/R3 并行前后做命名与编号一致性对齐；冲突时以上游为准。 |
| 证据来源不可得 | 部分信号无法落地 | RS-04 标注为不可得/需替代，交 RS-06 汇总，由调度方决定取舍。 |
| 研究越界产出设计/实现 | 侵占 design-planning 边界 | 单任务验收禁止产出数据模型/交互/实现方案，仅记录研究所得与开放问题。 |
| 预填研究结论 | 违反“不执行研究”约束 | 本计划仅列研究问题与调度方式，结论由执行阶段产出。 |

## 10. 调度建议与移交记录

- 调度顺序见第 8 节；并行批次见第 4 节（R1→R2→R3→R4）。
- 建议以 RS-06 的 design-planning 移交包（确认项 / 开放问题 / 阻塞项）作为进入设计阶段的前置门槛。
- 无待处理 review item（`spec.yaml` `review.items` 为空）。

交付路径：`plans/research-plan.md`。
