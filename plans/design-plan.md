# 任务质量仪表盘详细设计任务规划

## 1. 规划边界

本计划用于调度“spec-executor 任务质量仪表盘”项目的详细设计任务。规划输入为 `inputs/project-brief.md` 和 `docs/HLD.md`：项目需要先研究用户需要看的质量信号，再设计数据模型和交互，最后实现 CLI 导出和 Markdown 报告；当前任务只产出 `plans/design-plan.md`，不编写详细设计正文，不执行研究、设计或开发本身。

## 2. 设计任务拆分

| 任务 ID | 任务名称 | 详细设计输出建议 | 设计职责 | 验收标准 |
| --- | --- | --- | --- | --- |
| DD-01 | 质量信号与指标口径设计 | `docs/design/quality-signals-design.md` | 定义仪表盘需要展示的任务质量信号、指标口径、来源证据、缺失数据处理和质量分层规则。 | 覆盖“用户需要看哪些质量信号”；每个指标说明数据来源、计算口径、展示用途、不可用状态和后续任务输入。 |
| DD-02 | 数据模型与导出契约设计 | `docs/design/data-model-export-contract-design.md` | 设计任务、运行、验证、评审项、质量信号、汇总报告等数据对象，以及 CLI 导出与 Markdown 报告的数据契约。 | 覆盖“设计数据模型”和“CLI 导出、Markdown 报告”的数据基础；字段来源可追溯到 DD-01 或项目输入；包含序列化边界和兼容策略。 |
| DD-03 | CLI 流程与报告生成设计 | `docs/design/cli-report-generation-design.md` | 设计 CLI 输入输出、命令流程、读取/聚合/导出步骤、Markdown 报告结构、错误处理和验收命令建议。 | 覆盖“实现 CLI 导出和 Markdown 报告”；流程说明包含成功路径、失败路径、退出码、文件输出和验证方式。 |
| DD-04 | 仪表盘交互与信息架构设计 | `docs/design/dashboard-interaction-design.md` | 设计仪表盘视图、筛选/排序/钻取、质量状态呈现、空状态和用户阅读路径。 | 覆盖“任务质量仪表盘”的交互与展示；每个视图说明使用的数据对象、质量信号映射、关键状态和用户决策支持。 |
| DD-05 | 验收、风险与实施边界设计 | `docs/design/acceptance-risk-boundary-design.md` | 汇总详细设计验收策略、跨文档一致性检查、风险、停止条件、非目标和实施边界。 | 明确设计完成判定、实现前置条件、阻塞项处理方式；不引入超出 brief/HLD 的产品事实。 |
| DD-06 | 详细设计总体验收 | `docs/design/design-acceptance-report.md` | 对 DD-01 至 DD-05 的产物做最终一致性验收，确认是否可进入开发计划。 | 所有设计文档存在且互相一致；当前阶段需求均被覆盖或显式阻塞；无代码、测试、manifest、schema、迁移或运行时脚本修改。 |

## 3. 依赖 DAG

| 依赖 | 原因 |
| --- | --- |
| DD-01 -> DD-02 | 数据模型必须基于已确认的质量信号、指标口径和证据来源。 |
| DD-01 -> DD-04 | 交互设计必须知道用户要查看的质量信号及其状态含义。 |
| DD-02 -> DD-03 | CLI 导出和 Markdown 报告需要稳定的数据对象与导出契约。 |
| DD-02 -> DD-04 | 仪表盘视图需要使用统一数据模型，避免展示字段与导出字段分裂。 |
| DD-03 -> DD-05 | 验收设计需要包含 CLI 和报告生成的验证策略。 |
| DD-04 -> DD-05 | 验收设计需要包含仪表盘交互与展示状态的检查策略。 |
| DD-01 -> DD-05 | 风险与边界需要覆盖质量信号定义不足、指标歧义和缺失数据情形。 |
| DD-02 -> DD-05 | 风险与边界需要覆盖数据契约、兼容性和字段来源。 |
| DD-05 -> DD-06 | 最终验收必须先有统一的验收、风险与实施边界设计。 |

DAG 节点：DD-01、DD-02、DD-03、DD-04、DD-05、DD-06。  
环路检查：无环；DD-06 是唯一最终验收节点。

## 4. 并行策略

| 批次 | 可并行任务 | 调度条件 | 并行注意事项 |
| --- | --- | --- | --- |
| P1 | DD-01 | 读取 `inputs/project-brief.md` 与 `docs/HLD.md` 后启动。 | 先完成指标口径，避免后续设计凭空定义字段或视图。 |
| P2 | DD-02、DD-04 | DD-01 通过单任务验收后启动。 | DD-02 负责数据契约，DD-04 负责交互；两者可并行，但必须在交付前对质量信号命名和状态枚举进行一致性检查。 |
| P3 | DD-03 | DD-02 通过单任务验收后启动。 | CLI/报告设计不得绕过 DD-02 自行新增字段；新增需求必须回写为 DD-02 的设计问题。 |
| P4 | DD-05 | DD-03 与 DD-04 均通过单任务验收后启动。 | 汇总前必须检查 DD-01 至 DD-04 的输出路径和设计边界。 |
| P5 | DD-06 | DD-01 至 DD-05 全部完成后启动。 | 只做最终设计验收和交付记录，不补写详细设计正文。 |

## 5. 单任务验收要求

每个详细设计任务的验收必须满足：

- 只产出对应的 `docs/design/*.md` 设计文档或最终验收报告。
- 明确来源引用，至少覆盖 `inputs/project-brief.md` 和 `docs/HLD.md` 中与本任务相关的内容。
- 说明当前任务的输入、输出、控制流、数据流、错误/缺失数据处理、验证建议、风险和移交说明。
- 不修改生产代码、测试、构建配置、schema、迁移、运行时脚本或 task package 文件。
- 不把待研究内容、待确认产品事实或实现细节伪装成已确认需求；无法确认时记录为开放问题或阻塞项。
- 与上游任务的命名、字段、状态、验收口径保持一致；冲突时停止并要求修订上游设计。

## 6. 全局验收要求

DD-06 的最终验收需要确认：

- DD-01 至 DD-05 的所有计划文档均已存在，且文件路径符合规划。
- “质量信号研究 -> 数据模型和交互设计 -> CLI 导出和 Markdown 报告实现准备”的链路完整。
- 每个当前阶段需求都有设计覆盖、显式延期说明或阻塞说明。
- 依赖关系与实际文档引用一致，无隐藏依赖、循环依赖或重复职责。
- 并行批次执行后没有字段命名、状态枚举、报告结构或展示语义冲突。
- 验收证据可以支持后续 development-planning，但不包含详细设计以外的实现交付。

## 7. 停止条件

任一任务遇到以下情况必须停止并记录阻塞原因：

- `inputs/project-brief.md` 或 `docs/HLD.md` 缺失，或其内容不足以支撑该任务的设计边界。
- 需要新增项目未确认的用户画像、指标、数据字段、CLI 参数、报告格式或运行环境事实。
- 上游任务未完成、验收失败，或上游输出与当前任务目标冲突。
- 任务需要修改代码、测试、manifest、schema、迁移、运行时脚本、task package 或其他非设计文档文件。
- 详细设计任务无法被隔离成单一文档产物，或与其他任务出现职责重叠且无法通过依赖调整解决。
- 最终验收发现任一设计文档缺失来源引用、缺失验收策略、包含实现报告，或未说明风险和开放问题。
